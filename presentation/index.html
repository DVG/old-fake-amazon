Straight to the Demo

Can anyone think why this test wasn't a very good one?

  • Maintenance, everything is hardcoded
  • Doesn't actually test anything (no expectations set)
  • Repeated Steps
  • Relies on "known data", which can change anytime
  • Doesn't report any results
  • No fault tolerance, if anything goes wrong, the script just dies
  • No one except the tester has the first idea what the test actually does

Does anyone have any ideas on how to mitigate this?

This script was completely typical

  • Go into any enterprise test automation area, and you'll see hundreds of scripts just like this
  • Testers aren't coders, and aren't trusted to have access to the application, so they are forced to make do with an 'outside in' approach like they would manual tests in a test environment

Tests are Code

So you should know a little something about code

  • As soon as you start making tests, you're a programmer.
  • Programmers need to be clever to mitigate these problems
  • Programmers need useful tools

Object Oriented Programming (like Ruby!)

  • Encapsulation
  • Inheritance
  • Polymorphism

Encapsulation

  • Information Hiding - hides implementation, exposes interface

Inheritance

  • The behvior and implementation of the parent are inherited by the children

Polymorhism

  • Different objects may respond to the same message in different ways
